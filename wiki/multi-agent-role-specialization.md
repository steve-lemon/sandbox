# 멀티 에이전트 역할 분담 패턴

하나의 채팅 창·모델이 계획, 코딩, 검토, 도구 사용을 전부 처리하도록 만드는 "싱글 모델 함정(single-model trap)"에서 벗어나, 각 역할에 가장 적합한 모델/도구를 배정하는 워크플로우 패턴이다.

## 핵심 아이디어

모델 하나에 모든 책임을 억지로 몰아넣기보다, 다음처럼 역할을 분리한다.

- **오케스트레이터**: 상시 실행되며 메모리·스케줄링·메시징·로컬 도구를 보유 (예: [[hermes-agent]])
- **메인 에이전트 두뇌**: 일반적인 앞뒤 조정과 도구 사용 처리 (예: OpenAI Codex)
- **코딩 전문가**: 제한된 코딩 작업만 위임받아 처리, 구독 기반으로 실행 (예: [[claude-code-subprocess-delegation|Claude Code]])
- **실행 환경**: 로컬 상시 가동 머신 (미니 PC, NUC 등)
- **원격 인터페이스**: 어디서든 접근 가능한 메시징 채널 (예: Telegram)

## 확장 사례: 멀티 모델 파이프라인

댓글에서 공유된 변형 패턴은 TDD + JIT + 수직 슬라이스(vertical slice) 원칙을 결합한 역할 분담이다.

1. 상위 모델(예: Opus)이 다음 작업 청크(chunk)를 정의
2. 다른 모델(예: GPT 계열)이 청크를 구현
3. 세 번째 모델(예: 경량 Sonnet)이 빠르게 리뷰
4. 한 단계/마일스톤의 모든 청크 완료 후, 최초 정의 모델이 전체 마일스톤 QA 검토
5. 반복

이 변형은 하나의 게시물 댓글에서 언급된 사례이며 검증되지 않은 개인 워크플로우다 (status: needs-update, 재현/검증 필요).

## 왜 효과적인가

- 로컬 저사양 LLM으로 전체 파이프라인을 대체하려는 시도는 속도 면에서 비효율적 — 역할별 최적 도구 사용이 더 낫다는 것이 원문의 핵심 주장.
- 충분한 오케스트레이터-코딩전문가 검증 사이클이 쌓이면, 오케스트레이터가 "언제/어떻게 코딩 작업을 라우팅할지"에 대한 패턴을 스킬로 축적하게 되어 시간이 지날수록 실행이 빨라진다는 주장이 댓글에 있음 (추측/미검증, inference).
